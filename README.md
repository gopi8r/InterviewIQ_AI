# InterviewIQ

A voice-based, AI-generated technical interview platform. Candidates register with their
skills and experience, go through a rules/policy gate, then complete a spoken interview where
questions are generated by an LLM (Groq/Llama 3.3), transcribed by Whisper, and scored for both
technical accuracy and communication. Results are visible **only** to admins, via a completely
separate admin portal.

This version rebuilds the frontend in **React** (via Vite) and fixes several session-handling
and admin-usability issues from the previous prototype.

## What's new in this version

- **React frontend** (Vite + React Router) instead of static HTML/vanilla JS
- **Three-page candidate flow**: Login/Register → Rules & Policy (with an Agree gate) → Test
- **No more auto-login**: auth tokens are kept in memory only (not localStorage), so opening the
  site or refreshing always asks for login again - fixes the "shows previous user's details" bug
- **Admin portal is a separate, unlinked route** (`/admin`) - there is no link to it anywhere on
  the candidate site
- **Admin also always requires a fresh login** - the dashboard never auto-opens from a stored session
- **Search + pagination** on the admin candidates table
- **Bulk PDF export by date range** - download a ZIP of every completed interview's PDF between
  two dates
- **Defensive score rendering** - a candidate with no score yet (or edge-case data) now always
  shows `0`/`Pending` instead of throwing a raw JS error popup

## Tech stack

| Layer | Technology |
|---|---|
| Frontend | React 18 + Vite + React Router |
| Backend | FastAPI |
| Database | MySQL |
| ORM | SQLAlchemy |
| Auth | JWT (kept in-memory on the frontend, not persisted) |
| Speech-to-Text | Whisper (local) |
| LLM | Groq API (Llama 3.3 70B) - question generation + scoring |

## How the candidate flow works now

1. **`/`** - Login or Register (register also collects skills + years of experience). No link to
   the admin portal exists on this page.
2. **`/home`** - After login, the candidate lands on a rules/policy page with Do's, Don'ts, and an
   evaluation-consent policy. The **Start Test** button stays disabled until the "I agree" checkbox
   is checked.
3. **`/test`** - The actual interview: each question is read aloud, the candidate records a spoken
   answer (shown via an animated avatar, not a raw transcript), and after the last question the
   candidate sees only a "Thank you, submitted for review" screen - never their score.

Because auth state lives only in React memory, refreshing the browser or reopening the site at
any point starts over at the login page - there is no stale session or leftover previous user data.

## How the admin flow works now

1. **`/admin`** - a separate login page, not reachable from any link on the candidate site.
2. **`/admin/dashboard`** - only reachable after a successful login on `/admin`; if you paste this
   URL directly without logging in, you're redirected back to `/admin`. Refreshing the browser
   also requires logging in again, since the admin token is not persisted either.
3. From the dashboard: adjust the number of questions per interview, search/paginate through all
   candidates, view a full per-question breakdown, download a single PDF, or download a ZIP of
   every completed interview's PDF within a chosen date range.

## Step-by-step setup

### 1. Prerequisites
- Python 3.11 or 3.12
- **Node.js 18+** (new requirement for the React frontend) - download from https://nodejs.org
- MySQL Server running
- ffmpeg on PATH (required by Whisper)
- A free Groq API key from https://console.groq.com

### 2. Backend setup (same as before)
```bash
cd backend
python -m venv venv
venv\Scripts\Activate.ps1        # Windows PowerShell
# source venv/bin/activate       # Mac/Linux

pip install torch --index-url https://download.pytorch.org/whl/cpu
pip install -r requirements.txt
```
Copy `.env.example` to `.env` and fill in your real `DATABASE_URL`, `JWT_SECRET_KEY`,
`GROQ_API_KEY`, and admin credentials.

```bash
python -m app.seed_admin
uvicorn app.main:app --reload
```
Confirm it's running at http://127.0.0.1:8000/docs

### 3. Frontend setup (NEW - React via Vite)
```bash
cd frontend
npm install
npm run dev
```
This starts the Vite dev server at **http://127.0.0.1:5173** by default.

If your backend runs somewhere other than `http://127.0.0.1:8000`, update `API_BASE` at the top
of `frontend/src/api/api.js`.

### 4. Try it out
- **Candidate**: http://127.0.0.1:5173 → Register → Rules/Policy page → agree → Start Test
- **Admin**: http://127.0.0.1:5173/admin → log in with your `ADMIN_EMAIL`/`ADMIN_PASSWORD`

## Future enhancement: proctoring (camera + tab monitoring)

`frontend/src/components/ProctoringMonitor.jsx` is a scaffold (not yet wired into the test flow)
for two planned features:
- **Camera monitoring** - a live webcam preview during the test, with room to later detect if no
  face (or multiple faces) are visible
- **Tab/window switch detection** - already listens for the browser's `visibilitychange` event;
  just needs a backend endpoint to log flags against the session

To activate: render `<ProctoringMonitor />` inside `TestPage.jsx`, add a
`POST /api/interview/session/{id}/flag` endpoint, and a `proctoring_flags` JSON column on
`InterviewSession` in `models.py`. See the comments at the top of that file for the full plan.

## Things you must configure

Same as the previous version - MySQL credentials, Groq API key, JWT secret, ffmpeg on PATH,
admin credentials in `.env` before running `seed_admin.py`. See `backend/.env.example` for the
full list.
